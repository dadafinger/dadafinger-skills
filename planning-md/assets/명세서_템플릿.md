### 🔑 섹션 계약 (작성 시 1회 확정 — 자동화의 핵심)

**섹션 경계는 사람이 md 작성 단계에서 한 번만 판단해 확정한다.** 이 판단을 여기서 끝내면 이후 HTML·Figma·QA가 추측 없이 그대로 소비한다(병목 해소).

- **섹션 단위 = 와이어프레임 기준 "큰 레이아웃/기능 요소"** (예: 헤더 액션, 속성 필터, 현황 테이블, 조건 상세, 통보 설정). 무엇을 한 단위로 볼지는 판단이 필요하다 — 자명하지 않다. 너무 잘게 쪼개면 흩어지고, 너무 뭉치면 누락된다. → 상세 기준 `figma-desc-spec` 핵심원칙 1.
- ❌ **안티패턴(거부)**: 여러 기능요소를 「UI 요소 / 상태·케이스 / 동작 정책」 같은 본문 블록명으로 섹션을 나누지 말 것. 이 블록들은 섹션을 나누는 축이 아니라 **각 섹션 본문 안의 구성**이다(아래).
- **4중 정합 — 이 N이 4곳에서 동일하다 (불일치 = 누락 신호):**

  | 위치 | 표현 |
  |---|---|
  | md (이 문서) | `### N · 기능요소명` |
  | 좌측 목업 | ①②③ 마커 |
  | HTML 산출 | `.area-no N` |
  | Figma Description | `Section N` |

- 각 섹션 헤딩에 `<!-- marker=① screen=목록 -->` 주석으로 매핑을 명시한다(렌더러가 추측 없이 읽음). `screen`은 통합 폼(생성/수정/복제 1 URL)에서 모드 구분용.

## 영역별 명세

> 섹션 = 기능요소(위 "섹션 계약"). 각 섹션 **본문은 「UI 요소 → 상태·케이스 → 제약 → 동작 정책」 구성**으로 쓴다(알람 명세 정렬·동작 정책 맨 뒤, 없는 블록 생략 가능). 제목엔 번호(①)를 붙이지 않는다 — 번호는 좌측 거터 셀이 표시. 상세 규칙 = `figma-desc-spec` 핵심원칙 2.
>
> - **UI 요소** = 라벨 없이 **바로 넘버링**으로 구성요소 전수 나열. 분류 필요하면 **컴포넌트명 헤더 + 그 아래 넘버링**.
> - **제약** = 구성요소 하위별. `단, …`=예외 케이스 / `유효성: …`=유효성 정의 / 특이 케이스=`~할 경우 …` 명시.
