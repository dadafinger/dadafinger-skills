# 매뉴얼 이미지 검수 스킬

docx 매뉴얼의 이미지(스크린샷·다이어그램)를 검수한다. 텍스트 워싱(en-manual-wash)은
본문만 처리하므로, 캡처 속 민감정보·한글 잔존은 이 스킬로 잡는다.

## 왜 트리아지가 기본인가

이미지를 LLM이 직접 보는 단계가 유일한 병목이다(412장 전수 = 35~40회 왕복, 수십 분).
자동 스캔(OCR+규칙)은 전체 3분 내에 끝나고, 실제 문제의 대부분은 플래그된 10~20%에
몰려 있다. 그래서 기본 흐름은 **자동 스캔 전수 + 플래그 이미지만 육안**이다.
전수 육안이 정말 필요할 때만(최종 제출 직전 등) `--full`로 진행한다.

## 실행 흐름

스킬 루트: `agent-shared/skills/manual-image-review/`

```bash
OUT=/tmp/<매뉴얼명>_review   # 작업 폴더(아무 곳이나)

# 1) 추출 + 매핑 + 품질 메타 (수 초)
python3 scripts/extract_and_map.py "<매뉴얼>.docx" $OUT

# 2) OCR — 백그라운드 권장 (413장 기준 2~3분)
swift scripts/ocr_vision.swift $OUT/word/media $OUT/ocr_results.jsonl "ko-KR,en-US"
#   일본어 매뉴얼: "ja-JP,en-US" / 국문 매뉴얼: "ko-KR,en-US"(한글 플래그는 무시하고 판단)

# 3) 트리아지 큐 생성 (수 초)
python3 scripts/analyze_ocr.py $OUT [--keywords 추가키워드,쉼표구분]

# 4) 캡처 유형 분류 (수 초) — 사전 준비용, screenshot 워크플로 매핑에 사용
python3 scripts/classify_captures.py $OUT
```

산출물(모두 $OUT):
- `stats.txt` — 참조 수/중복/저해상도/미사용/재사용 요약
- `review_list.txt` — 전체 이미지 순서별 체크리스트(섹션·앞뒤 문맥 포함)
- `triage_queue.txt` — **육안 검수 대상 + 사유** (핵심 산출물)
- `ui_subtitles.txt` — 화면 타이틀-부제 쌍 (부제 카피 오류를 텍스트로 빠르게 스캔)
- `findings_auto.md` — 자동 검출 표 스켈레톤 (육안 판정 칸 채워가며 사용)
- `capture_types.csv` — 이미지별 캡처 유형 분류 (FULL/CONTENT/MODAL/PANEL/INLINE/DIAGRAM/COVER/CLI/TOAST). 룰·매핑은 `references/capture-type-taxonomy.md` 참조

## 검수 카테고리 정책 (사용자별 선택)

검수 대상 카테고리는 매뉴얼 환경에 따라 다르게 잡는다. **호출 시점에 명시**하고, 명시 없으면 디폴트(아래 첫 옵션)를 적용한다.

| 정책 | A 워싱 누락 | B 한글 잔존 | C 제품 UI | D 배치·중복 | E 품질 | 적용 맥락 |
|---|---|---|---|---|---|---|
| **default-strict** | ✅ | ✅ | ✅ | ✅ | ✅ | 운영 매뉴얼 / 외부 제출 / 실서버 캡처. PII·NDA·공인 IP 등 보안 사안 포함 |
| **content-excluded** | ❌ (콘텐츠 영역 데이터 일괄 제외) | ✅ | ✅ | ✅ | ✅ | **테스트 서버 / 더미 데이터 캡처**. 콘텐츠 영역의 이름·이메일·자원명·IP 등은 더미로 일괄 간주, 내용 진위 구분 안 함. B는 표지·다이어그램·UI 라벨/툴팁만 본다(콘텐츠 한글 테스트데이터 제외) |
| delta-only | (선택) | (선택) | (선택) | (선택) | (선택) | 베이스라인 대비 변경/신규 이미지만. 다른 정책과 조합 |

`content-excluded` 정책 시:
- A 카테고리는 본 리포트에서 제거, 부록(통계만)으로 보관
- B 카테고리는 표지·다이어그램·UI 라벨/툴팁 한정 (콘텐츠 영역 한글 테스트데이터는 제외)
- 직원 ID 파생 리소스명/가공 이메일/사설 IP/가공 프로젝트명은 검수 대상 아님

## 4) 육안 검수 (트리아지 큐 기준)

1. `triage_queue.txt`의 이미지를 **본문 순서대로 10~12장씩** Read 도구로 읽는다.
   경로는 `$OUT/word/media/<파일명>`.
   단, **한 변이 2000px를 넘는 이미지가 배치에 1장이라도 섞이면 요청 전체가 거부**되므로
   (큐에 px가 표기됨) 큰 이미지는 먼저 축소본을 만들어 그쪽을 읽는다:
   `sips --resampleWidth 1600 복사본.png`
2. 배치마다 판정을 **즉시** findings 파일에 기록한다(긴 세션에서 컨텍스트가 요약돼도
   파일이 상태를 보존한다). 다 보고 나서 몰아 쓰지 말 것.
3. 판정 기준과 단골 패턴은 `references/known-issue-patterns.md`를 먼저 읽고 시작한다.
   OCR 플래그는 단서일 뿐 — 한글 검출이 '언어 변경 화면'처럼 의도적인 경우,
   jpg 플래그가 정상 다이어그램인 경우는 '문제 아님'으로 판정하면 된다.
4. `ui_subtitles.txt`를 통째로 훑어 타이틀과 부제가 안 맞는 화면(예: Flavor 페이지에
   스냅샷 부제)을 찾는다. 이건 이미지를 안 봐도 텍스트로 잡힌다.

`--full`(전수 육안) 요청 시: `review_list.txt` 순서로 전 이미지를 같은 방식으로 본다.
412장 기준 35~40배치임을 사용자에게 먼저 알리고 진행한다.

## 5) 리포트

발견 사항을 심각도-유형으로 분류해 매뉴얼 폴더에 MD로 저장한다
(파일명 예: `<버전>_이미지검수_<날짜>.md`).

```
## 요약 (심각도순 건수)
## A. 워싱 누락 — 실직원 이메일/실명/고객사명/공인IP/개인 경로
## B. 한글 잔존 — 표지/다이어그램/UI요소/테스트데이터 구분
## C. 제품 UI 문구 오류 — 매뉴얼이 아니라 제품 수정 대상임을 명시
## D. 본문-이미지 배치/중복 의심
## E. 품질 — 저해상도/잘림/깨짐
각 항목: | ID | 이미지파일 | 섹션 | 내용 | 권고 |
```

분류 시 판단 원칙:
- **개별 수정 vs 정책 판단 구분**: 직원 ID 파생 리소스명(bq-, th.oh- 등)이 수백 곳이면
  개별 항목이 아니라 "재캡처 정책 필요" 하나로 통합하고 대표 예시만 든다.
- **부분 마스킹 패턴 주의**: Name 열만 마스킹되고 ID/Email 열은 노출된 화면처럼,
  워싱이 시도됐지만 불완전한 경우가 가장 위험하다(검수 통과로 오인되기 쉬움).
- 캡처 일자가 유난히 오래된 이미지(예: 다른 캡처는 6월인데 1~2월)는 구버전 UI/한글판
  혼입 신호다. OCR 텍스트의 날짜를 근거로 적어준다.
- **베이스라인 비교**: 이전 버전 docx가 있으면 추가로 추출(같은 `extract_and_map.py`)해
  MD5 해시 diff로 변경/신규/삭제를 식별. Δ만 우선 육안하고 나머지는 자동 플래그 요약으로
  처리해 시간 절약. 결과는 `delta_vs_{ver}.txt`로 저장.

## 캡처 유형 분류 (사전 준비)

`scripts/classify_captures.py`로 이미지를 9개 유형(+UNKNOWN)으로 분류한다. 룰과
screenshot 워크플로 매핑은 `references/capture-type-taxonomy.md` 참조. 본 분류는
**향후 자동 캡처 시 동일 유형으로 재출력**하기 위한 사전 작업으로, 실시간 포탈 접근이
없어도 메타+OCR만으로 분류 가능하다. 검수 리포트의 마지막에 캡처 유형 통계 + 신규/변경
이미지의 유형 매핑을 부록으로 첨부한다.

## 의존성·폴백

- OCR은 macOS Vision(swift) 사용. swift가 없으면 OCR을 건너뛰고 형식 기반 플래그
  (jpg/초소형/저해상도/표지) + 전수 육안으로 전환하되, 소요 시간을 먼저 알린다.
- docx가 아닌 PDF 매뉴얼이면 `pdfimages`로 추출 후 2~3단계만 재사용.
