# 4.0.0 Description 작성 예시 (distilled)

> 실제 Product 4.0 spec에서 추출한 "좋은 Description" 패턴. 원본은 `[internal-repo-path]/_참고자료/spec-format/`(알람 4종·집계 호스트·복제). 작성 시 톤·입도(granularity)의 기준으로 삼는다.

## A. 목록 (List) — 알람 관리 v4 / `CONT-09_10_01`

컴포넌트 단위 섹션 + 각 섹션 본문 구성(UI 요소 → 상태·케이스 → 제약 → 동작 정책, 동작 정책 맨 뒤). 발췌:

- **메뉴/헤더**: LNB `[알람 센터]·[알람 이력]·[알람 관리](활성)` 3메뉴. 헤더 액션 = 활성·비활성·수신자 추가·수신자 제거·삭제·생성(**다건 선택 시 활성**). 관리자 전용.
- **속성 필터**: 대상 유형·심각도·활성 여부·기본 제공룰 패싯 + 이름·대상·수신자 키워드. **필터 초기화** 버튼·적용 칩·N건 카운트. 컬럼 공급자 → **Product 엔진**(값 통일). 목록 전체 로드(최대 200건).
- **일괄 작업**: 활성/비활성 = **탐지 시작/종료**(비활성해도 알람 센터 기발생 알람 영향 없음). **삭제 확인 모달**: "룰을 삭제해도 알람 센터의 발생 알람은 유지됩니다." 기본 제공 룰은 삭제 불가(제외 안내).
- **작업(⋯) 메뉴**: 수정 → `?mode=edit` / 복제 → `?mode=clone` (표로 동작 명시).
- **컬럼 전수**: 이름(+조건 수)·Product 엔진·지표·적용 범위·대상 유형·대상 상세·조건·심각도·통보 채널·활성(탐지)·기본룰·수신자 그룹(+개별)·작업.

**배울 점**: 헤더 액션의 "다건 선택 시 활성" 같은 **동작 조건**까지 적고, 확인 모달 문구를 **정확한 한국어 그대로** 박았다.

## B. 통합 폼 (생성·수정·복제) — 알람 수정 v3.5 / `CONT-09_10_02/06`

`?mode=create|edit|clone` 통합 폼 → **모드별 편집 정책 매트릭스**가 핵심:

| 섹션 | 생성 | 수정 | 복제 |
|---|---|---|---|
| 기본 정보 | 입력(공급자 조회전용) | 잠금 | 공백(공급자 조회전용) |
| 조건·조합·대상 | 입력 | 잠금 | 원본 프리필+수정 |
| 심각도 | 입력 | 잠금 | 입력 |
| Action | 입력 | 편집 가능 | 공백 |

- **잠금 정책 명시**(수정 모드에서 가장 누락 잦음): "공급자 = 조회 전용 — 진입 공급자명을 자물쇠 아이콘과 함께 표출, 수정 불가."
- 토글 의미: "활성화 = 탐지 시작 / 비활성화 = 탐지 종료(기발생 알람 영향 없음). 수정 모드 잠금."
- 정보구조 그루핑: 통보 = 통보 채널 / 수신자 그룹(BF 수신자 그룹) / 개별 수신자(BF 계정·이름(이메일)) 3층.

**배울 점**: 통합 폼은 **매트릭스 한 장**으로 모드별 입력/잠금/공백/편집가능을 못 박는다.

## C. 상세 (Detail) — 알람 관리 상세 v2.2 / `CONT-09_10_04`

- **헤더 + 버튼 동작 표**: 복제 → `?mode=clone`(조건·조합 원본 프리필+수정, 기본정보·Action 공백) / 수정 → `?mode=edit`(Action·수신자별 심각도만) / 삭제(기본 제공 룰 불가, 확인 모달 문구).
- 헤더 하단 **잠금 안내**: "수정은 Action·수신자별 심각도만, 기본정보·조건·조합·룰 기본 심각도는 잠김 → 변경은 복제."
- **좌측 기본 정보 전 필드 나열**: 이름·룰 ID(복사)·설명·공급자·심각도·활성 여부(탐지 의미)·기본 제공룰·대상 유형(복합 시 구성 목록)·생성자·생성/최종 수정 일시.
- **각 탭 전 섹션**: 상세 탭(조건 요약·조건 표·조합식·수정 불가 안내·심각도·권장) / 통보 설정 탭(배너·통보 상태·수신 그룹 표·분할 발송·채널·상세 URL).

**배울 점**: 상세는 **좌 기본 정보 필드 전수 + 각 탭 전 섹션 + 동선 URL**을 빠뜨리지 않는다.

## D. 데이터 화면 — 집계 보고서 호스트 / `MON-AGG-01_01_01`

영역별 명세를 `### 1`~`### n`으로 번호 부여(목업 라벨과 1:1). 데이터 화면 특유의 깊은 명세:

- **상태/모드 분리**: clean(적용 완료) vs dirty(편집 중) — 헤더 배경색 전환, diff pill 표기 형식 6종까지 표로.
- **빈 상태 3종**(드롭다운 결과 없음 / 선택 0건 / 선택 후 검색 결과 없음)을 **케이스·노출 위치·문구** 표로.
- **유효성 메시지**: step(hourly/daily/weekly/monthly) × 케이스(end<start, end=start, 정상)별 **정확한 메시지**를 표로 전수.
- **제약값**: 최대 대상 8개·데이터 포인트 200개·표 100행 등 cap을 **값·이유**까지.
- **인터랙션/동작 정책**을 `D-01`~`D-10` ID로 표 정리. 미결은 `T-/P-/U-` ID로 분리.

**배울 점**: 데이터 화면은 **케이스·메시지·제약값을 표로 전수**하고, 동작 정책에 ID를 부여해 추적 가능하게 한다.

## 공통 작성 톤 (4종 관통)
1. 동작 **조건**까지 적는다 ("다건 선택 시 활성", "수정 모드 잠금").
2. 사용자 노출 **문구는 정확한 한국어 그대로** 박는다(토스트·확인 모달·에러·빈 상태).
3. 케이스가 여러 개면 **표로 전수**(빈 상태·유효성·모드 매트릭스·diff·cap).
4. 화면 간 **동선 URL**(`?mode=...`)을 명시한다.
5. 보류는 **Open Issues**(`OI-N`)로 분리, 본문에 결론을 추측으로 채우지 않는다.
