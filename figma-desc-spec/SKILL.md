# Product Spec Description 작성 표준 (SoT)

상세 명세 페이지 **우측 Description 칼럼**(번호 섹션 + Appendix + Changelog)을 **무엇을 어떻게 쓸지** 정의하는 콘텐츠 표준. 이 스킬은 **내용·포맷 표준(WHAT)** 의 단일 출처(SoT)다.

> **역할 분담**
> - **이 스킬 = WHAT**: Description에 들어갈 내용·구조·서식 의미·전수 서술·검증.
> - **figma-change-flow = HOW**: Figma 노드에 실제로 그려넣는 기법(폰트 로드/버그 회피·`setFlat`(플랫 인코딩 정본)·`textAutoResize`·높이 정합·클론).
> - **planning-md/-html/-figma = 상위 + 렌더**: spec md 작성(섹션계약 `planning-md/assets/명세서_템플릿.md`) · HTML(`planning-html/assets/공통템플릿/`) · Figma 새 프레임. 연속 실행=planning-flow.

## 상세 명세 페이지 구조

```
┌─ 헤더행 ──────────────────────────────────────────────────────┐
│ Screen ID | 번호(CONT-XX_XX_XX) | Page Title | 페이지명 | Author | 작성자 │
├───────────────────────────┬───────────────────────────────────┤
│ 좌: 1920px 목업           │ 우: Description 칼럼 (704×2)        │
│  - ①②③ 영역 마커          │  ├ [Description] 검은 타이틀 바     │
│  - 빨강 변경 박스/모달 콜아웃 │  │   Intro(거터 없음 → 온점 넘버링) │
│                           │  │   1. <컴포넌트명> …               │
│                           │  │   2. <컴포넌트명> …               │
│                           │  └ [미결 사항] 본문서 분리한 OI 모음(맨 아래) │
└───────────────────────────┴───────────────────────────────────┘
   (현 4.0.0 기준: 일정·검증/경계 메모·[Changelog]는 개별 페이지 미기재 → 전역 트래커/자체 이력 페이지.
    미결 사항은 삭제하지 않고 페이지 맨 아래 「미결 사항」 블록으로 분리해 유지)
```

- **목업 ①②③ 마커 ↔ Description 번호 섹션은 1:1.** 마커 N개 = 섹션 N개. 개수 불일치 = 누락 신호.
- **🔴 목업 빨간 동그라미(①②③) = 좌측 mockup column 필수** — HTML `.area-no`와 동일 역할. Description 우측 거터(50px `1·2·3`)와 **번호 1:1·위치는 mockup 쪽**. **와이어프레임에 마커 없이 Description만 채우면 FAIL** — 서식 일괄(B)에서 `[node-id]`/`[node-id]`에서 clone 배치([`format-batch-pass.md`](../figma-change-flow/references/format-batch-pass.md)).
- Description을 **거대한 단일 텍스트 노드에 붓지 않는다** (가독성 지적 1순위). 섹션·불릿은 노드 단위로.
- **번호 표기 (사용자 확정 2026-06-23)**: 거터(좌측 번호 칼럼)가 **있는** 섹션은 **거터가 번호를 표시 → 제목·본문에 번호를 붙이지 않는다**(핵심 원칙 1). 거터가 **없는** 설명란(예: Intro, 거터 없는 보조 설명)은 **본문을 온점 넘버링(`1.` `2.` …)으로 시작**한다. 동그라미 번호(①②③)는 제목·본문 넘버링으로 쓰지 않고 **목업 영역 마커 참조용으로만**.

## 핵심 원칙 1 — 섹션 단위 = 와이어프레임 기준 "큰 레이아웃/기능 요소" (판단 필요)

번호 섹션(①②③)의 단위는 **와이어프레임을 보고 페이지를 큰 레이아웃 단위로 나눈 "기능 요소"** 다. 예: 액션 버튼 영역, 속성 필터, 현황 테이블, 조건 상세, 통보 설정.

- **와이어프레임의 큰 레이아웃 블록**을 기준으로, 사용자가 하나의 기능으로 인지하는 묶음 단위로 끊는다. 화면 내에서 요소 3~10개 이하 요로 분할해 작성한다.
- 이 섹션 단위가 곧 목업 마커(가터)이며, 목업 마커 N개 = Description 섹션 N개 로 1:1 매핑된다(검증 신호).
- 번호는 좌측 거터 셀이 표시. 제목은 기능 요소명만.
- 한 번 넘버링 및 작성한 설명은 수정시에 버전관리를 위해 질문한다.
- 하위 분할 번호 사용은 지양한다.(e.g.3-1)
- ❌ **안티패턴(사용자 명시 거부)**: 여러 기능 요소를 「UI 요소 / 상태·케이스 / 동작 정책」 같은 본문 블록명으로 섹션을 나누기. (이 블록들은 섹션을 나누는 축이 아니라, **각 섹션 본문 안의 서술 구성**이다 — 핵심 원칙 2.)
- ✅ 기능 요소별로 섹션을 나누고, **각 섹션 1개의 본문을 「UI 요소 → 상태·케이스 → 제약 → 동작 정책」 구성으로** 쓴다.

## 핵심 원칙 2 — 섹션 본문 구성 (알람 명세 정렬: UI 요소 → 상태·케이스 → 제약 → 동작 정책)

각 기능요소 섹션의 본문은 **알람 기획서와 동일한 구성 블록**으로 쓰되, **순서는 「동작 정책」을 맨 뒤**에 둔다(사용자 확정 2026-06-23). 해당 블록이 없으면 생략 가능.

| 순서  | 블록         | 라벨 표기                     | 내용                                               |
| --- | ---------- | ------------------------- | ------------------------------------------------ |
| 1   | **UI 요소**  | **라벨(제목) 없이 바로 넘버링으로 시작** | 그 섹션의 구성 요소(필드·옵션·컬럼·버튼·영역)를 번호 리스트로 전수 나열 + 설명. |
| 2   | **상태·케이스** | `상태·케이스`                  | 컴포넌트의 상태/케이스 정의(예: clean/dirty, 빈/부분/오류).        |
| 3   | **제약**     | `제약`                      | **UI 요소(구성요소) 하위별로** 작성. 표기 관례 ↓.                |
| 4   | **동작 정책**  | `동작 정책`                   | 섹션 전체의 동작·규칙(**맨 뒤**).                           |

- **UI 요소 = 라벨 제거·바로 넘버링**: 본문 첫 블록은 「UI 요소」 헤더를 적지 않고 곧장 `1. 2. 3.` 넘버링으로 구성 요소를 연다.
- 구성 요소가 많아 분류가 필요할 때는 **컴포넌트명을 상단 구분 헤더**로 두고 그 아래에서 하위 넘버링한다(분류 불필요하면 단일 넘버링).
- **제약 표기 관례 (구성요소 하위별로 추가 작성)**:
  - `단, …` 로 시작 → **예외 케이스** 정의.
  - `유효성: …` 로 시작 → **유효성(Validation)** 정의.
  - 특이 케이스 → **`~할 경우 …`** 형태로 조건을 **명시적으로** 안내한다.
- 「동작 정책」을 맨 뒤로 두는 이유: 독자가 화면 구성요소(UI 요소)부터 인지한 뒤 상태·제약을 거쳐 **마지막에 동작 규칙**을 읽게 한다 — 알람 명세와 블록 셋은 동일, 순서만 조정.

> **마이그레이션**: 이전 표준의 「정책 / 항목(옵션 및 설명) / 제약·케이스」 3단은 위 구성으로 대체된다 — 정책 → **동작 정책(맨 뒤)**, 항목(옵션 및 설명) → **UI 요소(라벨 제거·넘버링)**, 제약·케이스 → **상태·케이스 + 제약**(분리).

## 핵심 원칙 3 — 서식 = 섹션 본문의 플랫 인코딩 마커 (정본 B)

> ⚠️ 계층 정리: **최상위는 "섹션"(핵심 원칙 1 — 와이어프레임 큰 레이아웃 기능 요소, 판단 필요)** 이며, 섹션 제목은 거터 번호 + title-bar 텍스트로 리스트가 아니다. 아래 깊이는 **그 섹션 "본문" 안에서만** 적용된다.

섹션 본문은 **플랫 인코딩 마커**(진짜 글자/공백)로 **한 TEXT 노드**에 표현한다 — 렌더는 `figma-change-flow` `setFlat`. 계층 마커·들여쓰기(구 네이티브 리스트 자동 넘버링 A는 폐기):

- **L0 소제목** `■ `(공백0) = 본문 블록 라벨 `상태·케이스`/`제약`/`동작 정책`. **단 「UI 요소」 블록은 소제목 없이 L1부터** 바로 시작(핵심 원칙 2).
- **L1**(공백2) = 항목. **컴포넌트·분기(열거) = 번호 `1. 2. 3.`, 단순 정책·제약(진술) = 글머리 `• `.**
- **L2**(공백5) = 세부/분기. 열거 = `a. b. c.`(그룹마다 리셋), 진술 = `• `. **L3**(공백8) = `• `.
- 마커·공백을 **직접 글자로 찍는다**(렌더러 자동 넘버링 아님). `a.` 카운터도 글자로 리셋. 상세 = 위 정본 표 + `setFlat` 상수.
- **🟢 항목별 꼼꼼히 = 가독성 우선 (사용자 확정 2026-06-29, 손본 예시 `[node-id]` CONT-07_04_01 미터링 이력)**: 「UI 요소」 항목과 「상태·케이스/제약/동작 정책」 각 항목을 **한 줄에 하나씩, 세부는 한 단계 들여써** 전수 나열한다. 여러 옵션·조건·사실을 **한 줄/한 문단에 압축(lumping)하지 않는다** — 한 줄에 몰아넣으면 읽기 어렵다(가독성 지적 1순위). 컴포넌트가 여러 옵션을 가지면 컴포넌트명을 상단 라인으로 두고 그 아래 옵션을 각각 별도 라인·들여쓰기로(예: `조회 조건` → `삭제 VM 포함 (기본 ON)` / `기간 (퀵 24h/7d/30d/사용자 지정)` / `인스턴스 — 프로젝트 선택 시 하위 일괄 선택`). 「항목/동작 정책/제약」 블록은 필수로 두되, 그 안을 이렇게 **항목 단위로 잘게** 쓰는 것이 핵심.
- **🟢 동일 정의·동작 = 완전 동일 문구 유지 (사용자 확정 2026-06-29)**: 같은 정의, 같은 제약, 같은 동작 정책은 화면이 달라도 **의도적으로 동일 문장**을 반복 사용한다. 추후 정책 변경 시 검색 후 일괄 치환할 수 있도록 동의어·축약·어순 변경을 피한다. 화면별로 달라지는 값만 변수처럼 치환한다(예: `호스트` ↔ `인스턴스(VM)`, `{host}` ↔ `{vm 또는 project}`); 공통 문구인 `조건(기간·대상·지표) 변경 시 표·차트 초기화 → [적용] 재조회 시 갱신 · 변경 강조`처럼 동일 의미의 정책은 토씨까지 맞춘다.
- **이모지/픽토그램 글리프 지양 (사용자 확정 2026-06-18)**: 본문에 `💾 ▷ ↶ ⌬ ✓ ● △ ☐ ⓘ ▼ ▶` 같은 이모지·장식 글리프를 쓰지 않는다(캡처·제출 시 폰트 깨짐·톤 불일치). 대신 **의미 있는 표기 관례만** 사용: 버튼=`[버튼명]`, 브레드크럼/뎁스=`[상위 > 하위]`, 목업 영역 마커=`①②③`, 라벨 인용=`「」`, 흐름=`→`, 구분=`·`. `↑`(ditto)·`─`(indeterminate) 같은 기호도 말로 풀어쓴다(예: `이하 반복 생략`, `부분 선택 시 indeterminate 표시`).
- **본문 내 표(table) 지양 (사용자 확정 2026-06-24)**: Description은 리스트로 서술한다. 표는 **정말 중요한 경우에만** 예외적으로 사용하고, **확정되지 않은 정의(예: leaf/ID·단위·산출식 추정 표)는 표로 넣지 않는다** — 추정값을 확정 사실로 단정해 정합성을 깨뜨리는 주원인([[feedback-prototype-mock-values]]). 기존 미확정 표는 제거하고 리스트로 대체.
- **카테고리/모드 선택 등 비기능 리드인 섹션은 거터 번호 생략 (사용자 확정 2026-06-24)**: 카테고리 탭처럼 기능 요소가 아닌 최상단 모드 선택 섹션은 `0` 같은 번호를 붙이지 않고 거터를 비운다. 기능 섹션은 `1`부터, 필터 하위 조건은 `1-1/1-2/1-3`.

### 핵심 원칙 2·3 — 2026-06-30 확정 (구성요소 넘버링·중첩·표기 보강)

집계 호스트 케이스(`[fileKey-removed]`) 재구성에서 확정. 위 원칙 2·3을 **구체 렌더 규칙으로 못 박은 것**.

- **본문 = 구성요소 넘버링 + 정책 중첩** (🔒 정본 = 플랫 인코딩 B, `figma-change-flow` `setFlat`):
  | 깊이 | 선두 공백 | 열거형(컴포넌트/분기) | 진술형(정책/제약/세부) |
  |---|---|---|---|
  | L0 소제목 | 0 | — | `■ ` |
  | L1 | 2 | `1. 2. 3. ` | `• ` |
  | L2 | 5 | `a. b. c. ` (그룹마다 리셋) | `• ` |
  | L3 | 8 | — | `• ` |
  - **제약·동작 정책은 관련 구성요소 "하위"로 중첩**(원칙 2의 「제약 = 구성요소 하위별 작성」을 동작 정책까지 확장). 별도 말미 `동작 정책` 헤더 블록 대신, 섹션 전체 동작은 말미 구성요소(예: `필터 영역 구조`·`표 구조`·`레이아웃`) 아래에 둔다 — 「동작 정책 맨 뒤」의 위치 의도는 유지, 표현만 구성요소 하위 중첩으로.
  - **번호 vs 글머리 분기**: 컴포넌트·분기(열거) = 번호(`1.`/`a.`), 단순 정책·제약·세부(진술) = 글머리 `•`. (컴포넌트=번호 / 정책=글머리, 2026-07-01 확정)
  - **렌더 = 진짜 글자/공백만** — 노드 전체 `NONE`/indent0/Noto Sans Regular로 1회 정규화, 이후 `characters=` 덮어써도 무손상. 들여쓰기 U+0020만(탭·전각 금지). 글머리 `•`(U+2022)는 인라인 구분자 `·`와 다른 글자. ⚠ 구 A(네이티브 `setRangeListOptions`+indent)는 폐기 — 깨짐 원인.

- **거터 평면 넘버링 = 전 UI 유닛 1~N (시각 순서)**: 섹션(유닛)마다 거터에 연속 번호. **본문 동그라미 참조(②③⑤⑥…)는 거터 번호와 동기화** — 거터 재번호 시 본문 참조도 **원본 기준 동시 치환**으로 재매핑(순차 치환 금지). 빈 거터는 기존 num 텍스트 노드 `clone()`으로 복원. (집계 호스트=`1~8`. ⚠️ 이때 `카테고리`도 `1`로 번호함 — 06-24 「카테고리 거터 생략」과 충돌 시 **유닛 전수 번호 우선**, 케이스별 사용자 확인.)

- **🔴 표기 관례 보강 (사용자 확정 2026-06-30)** — 위 원칙 3 「표기 관례」 라인을 갱신:
  - **항목명↔설명 연결 = 중간 대시 `—`** (em dash). `·`(가운뎃점) 같은 특수문자를 연결자로 쓰지 않는다. 항목명 뒤(또는 줄바꿈 뒤) 설명을 이 대시로 잇고 **온점(`.`) 없이** 끝낸다. (`·`는 한 줄 안의 나열 구분으로만.)
  - **안내 문구·메시지 = 큰따옴표 `" "`**. 토스트·검증 안내 등 사용자에게 보이는 문장은 `"조회기간을 먼저 선택해 주세요"`처럼 `""`로 인용. (`「」`는 라벨/버튼/UI 명칭 인용에만; 문구·메시지는 `""`.)
  - **`VM` 표기 금지 → `인스턴스`**: 솔루션 내에서 `VM`을 쓰지 않는다. `인스턴스(VM)`·`VM은…` → `인스턴스`·`인스턴스는…`로 대체.
  - **거터 옆 Title(title-bar 텍스트) = "영역 기능명" 중심** (UI 요소 나열 아님). 예: `카테고리 Tab`, `필터 영역`, `데이터 테이블`. 구성요소 전수 나열은 본문 넘버링이 담당하고, 제목은 그 섹션의 영역·기능을 한마디로.

- **🟢 통제 어휘집 (controlled vocabulary)** → [references/용어집.md](references/용어집.md): 화면마다 동일 대상에 동일 용어를 쓰기 위한 표준. UI 컴포넌트 타입(`base.css` 기반: 버튼·칩·토글·체크박스·셀렉트·테이블·모달·노트…)·영역 기능명·상태값·액션 용어·표기 관례를 정의. **미결/참조에 약어 토큰(`U-10`·`F-16`) 금지 → 짧은 한글 주제어 사용**(이 표준의 §6). 신규 작성 시 구성요소·영역명을 여기에서 가져온다. (3.0.6 PDF 기존 용어 심화 추출은 다음 패스.)

## 핵심 원칙 4 — 문체 = 정책서 톤 · 명사형(개조식) 종결 (사용자 확정 2026-06-23)

Description 본문은 **정책서와 같은 결**로, **사람이 직접 쓴 자연스러운 문장**으로 쓰되 **명사형(개조식)으로 종결**한다.

- **명사형 종결**: 문장을 `~함 / ~음 / ~할 것 / ~없음` 또는 순수 명사구로 끝낸다. 경어체(`~합니다`)·평서 종결어미(`~한다`/`~된다`)를 본문 서술의 기본 종결로 쓰지 않는다.
  - ✅ `카테고리 전환 시 ② 지표 트리 자동 정합 재계산` · `VM은 프로젝트 선택 선행 필수, 미선택 시 「프로젝트를 먼저 선택하세요」 안내` · `대상 한도 없음`
  - ❌ `카테고리를 전환하면 지표 트리가 자동으로 재계산됩니다.` · `재계산된다.`
- **사람이 쓴 톤**: AI 정형구·군더더기 접속(`또한`, `뿐만 아니라`, `~하는 것이 중요합니다`)·과한 수식 지양. 정책서처럼 간결·단정.
- 라벨·버튼·흐름 표기 관례는 핵심 원칙 3과 동일(`[버튼]`·`「라벨」`·`→`·`·`). 표기만 명사형으로 통일.

## 검증 룰 (작성·수정 후 필수)

1. **목업 ↔ Description 1:1 매핑**: 좌 목업 ①②③ 마커 개수 = 우 Description 섹션 개수.
2. **체크리스트 전수 확인**: [screen-type-checklists.md](references/screen-type-checklists.md)의 모든 항목을 명시했는가 (또는 "N/A 사유" 명기).
3. **4영역 확인**: 화면 분기·예외 6종·Validation 5요소·동적 컴포넌트 동작·안내 팝업/토스트 문구가 정확한 한국어로 적혔는가.
4. **Changelog 비어 있지 않은가**, 변경 마커 형식·색상 일치하는가.

### 누락 발견 시 처리 (추측 금지)
- **명세 없음** → 기획자에게 의도 확인 후 추가.
- **회의록엔 있고 명세엔 없음** → 회의록 인용 후 반영, Changelog에 출처 명시.
- **일정·검증/경계 메모** → Description에 남기지 않는다(제거 또는 전역 트래커 이관).
- **결정 보류(미결)** → 본문(Description 섹션)에 남기지 말고 **페이지 맨 아래 「미결 사항」 블록으로 분리**(아래 규칙).

### 미결 사항(Open Issues) 분리 규칙 (사용자 확정 2026-06-23)

- **본문에서 제외**: 섹션 본문은 **현행 사양만** 기술하고, `(미결)`·결정 대기 항목을 본문에 인라인으로 남기지 않는다.
- **위치 = 맨 아래**: 모든 미결을 페이지 **최하단 「미결 사항」 블록**(있으면 Appendix 다음, Changelog 운영 시 그 앞)에 한데 모은다. **삭제하지 않는다 — 일정·검증 노트와 달리 미결은 분리해 유지**(사용자 확정 2026-06-24).
- **🔴 지표 트리 등 본문에 섞지 않는다**: 미결 상세(후보 지표·결정 대기 항목)를 ② 지표 트리 같은 현행 사양 표/본문에 끼워 넣지 않는다 — 반드시 맨 아래 미결 블록으로 분리(실사례 2026-06-24: 인스턴스 지표 트리에 잘못 들어간 미결 행 일괄 제거).
- **🔴 신규 넘버링 금지**: `[OI-N 결정 대기]` 같은 새 번호 체계를 도입하지 않는다(사용자가 쓰지 않는 표기). **기존 화면 식별자 토큰**(예: `U-10`, `T-11`, `V-12`, `F-16/F-17`)을 그대로 유지해 항목 머리에 단다.
- 각 미결 행 = `토큰 (미결) — 한 줄 쟁점` 형태 — **토큰 바로 뒤에 「(미결)」을 붙여 미결 상태를 명시**한다(사용자 확정 2026-06-24). 명사형 종결(아래 핵심 원칙 4).
  - 본문 제약 등에 미결을 인라인으로 둘 수밖에 없을 때(블록 미분리 페이지)도 동일하게 `토큰 (미결) — …`로 표기한다.
  - `OI-3`/`OI-4` 같은 기존 Open Issues 토큰도 그대로 두되 동일하게 `(미결)`을 단다.
- 결정되면 본문(현행 사양)으로 승격하고 미결 블록에서 제거(「(미결)」 표시도 함께 제거).
