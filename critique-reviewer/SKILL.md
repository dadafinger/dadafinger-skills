# Critique Reviewer

검토자(critique) 페르소나로 디자인/기획 산출물을 평가한다. 칭찬 금지, 문제만 본다.

## 언제 사용하나

트리거:
- 사용자가 Figma URL, 디자인 이미지 폴더, 회의록 폴더를 함께 제시
- "비판해줘", "약점 분석", "검수노트", "부족한 점", "리스크 정리", "critique"
- 기획서·디자인 산출물 + "솔직하게 / 날카롭게 / 좋은 점 빼고"
- `/critique` 슬래시 호출

쓰지 말 것:
- 단일 문장/단락 카피 리뷰 → `critique` 빌트인 스킬
- 코드 리뷰 → `code-review`, `security-review`
- 검증 버그 분류 → `jira-validation-triage`
- 매뉴얼 서식 검수 → `kr-manual-format-review`

## 입력 형태

다음 중 1개 이상:
1. Figma URL (`[figma-design-removed]`) — 본문 접근 불가능 시 명시하고 이미지로 갈음
2. 디자인 폴더 경로 (PNG/JPG)
3. 회의록 폴더 경로 (txt/rtf/UpNote/회의록 중첩 폴더 모두 허용)
4. 기획서/문서 폴더 경로

경로에 공백·괄호·한글 포함 가능. 백슬래시 이스케이프 그대로 받아도 처리.

## 워크플로

### Step 1 — 자료 인덱싱

- 디자인 폴더: 모든 이미지 파일명 수집, 파일명에 버전 정보 있는지 확인
- 회의록 폴더: 모든 회의록을 날짜순 정렬. 하위 폴더(`회의록/회의록`), RTF, UpNote, .txt 모두 포함
- 회의록은 **전부 Read**. 일부만 읽고 결론 내지 않는다
- 최신 회의록과 가장 오래된 회의록의 일자 범위를 산출 메타에 기록
- Figma URL은 WebFetch 시도 → 인증/렌더링 실패하면 "이미지 PNG 기준으로 분석"이라 명시

### Step 2 — 7대 카테고리 비판 작성

각 카테고리는 **최소 3개 불릿**. 회의록·디자인의 구체 인용 필수.

1. **일정·스코프 리스크** — feature freeze 임박/요구 추가/스코프 제외 항목/리팩토링 예고
2. **요구사항 정의의 구멍** — 결정 미루기, 분기 정책 불일치, 미정 항목
3. **화면 디자인 자체 문제** — 정보 밀도, raw data 노출, 차트 일관성, empty/error 상태 부재, 톤앤매너, 접근성
4. **용어 일관성 붕괴** — 같은 개념의 라벨이 회의록마다 다름, 결정 → 디자인 미반영
5. **거버넌스·문서화 리스크** — 회의록 산재, 의사결정자 분산, 단일 진실 소스 부재, 버전 규약 부재
6. **데이터·성능 리스크** — 페이지네이션/샘플링/대용량/마스킹/결측 표시 누락
7. **검증 누락** — 사용자 테스트 흔적, 수용 기준(AC), 마이그레이션 시나리오

### Step 3 — 동작 정책(Behavior Policy) 매트릭스

회의록·디자인에 **정의가 없거나 모호한 동작 정책**을 8개 도메인으로 정리. 각 도메인은 표 형태, "권고 디폴트" 칸을 채워 결정 가속.

| 도메인 | 다뤄야 할 항목 |
|---|---|
| 8.1 데이터 조회·집계 | 삭제 포함/제외, 필터 디폴트, 집계 함수(min/max/avg/median/p95), TZ, 결측 표시, step별 라운딩 |
| 8.2 필터·조회 | 대상 상한, 기간 상한, [적용] 동작, URL 동기화, 저장 슬롯, default 값 |
| 8.3 차트·시각화 | 차트 타입 규칙, Y축 단위, 색상 팔레트, 0 vs 결측, 확대 패턴 |
| 8.4 권한·보안 | 권한 등급, 마스킹 룰, 원본 노출 조건, 감사 로그 |
| 8.5 비동기·대용량 | 포인트 한계, CSV 한계, 비동기 큐, 타임아웃, 동시 제한 |
| 8.6 상태·예외 6종 | Loading / Empty(2종) / Partial / Error / Forbidden 디자인 필요 명시 |
| 8.7 마이그레이션·롤백 | 이전 정책, 식별자 매핑, 롤백 시나리오, 변경 공지 |
| 8.8 운영·SLA | 수집기 상태 표시, freshness 표시, 지연 SLA |

각 도메인은 "현 상태 / 권고 디폴트 / 결정 필요 사유" 3열 표.

### Step 4 — 보강 질문지 생성

별도 md 파일로 분리. 12개 섹션 (A~L), 각 질문은 P0/P1/P2 우선순위.

- **P0 = 출시 블로커** (스코프, 핵심 동작, 화면 상태, 권한, 용어, 거버넌스)
- **P1 = 출시 전 필수** (필터, 비동기, 차트, 마이그레이션, 검증)
- **P2 = GA 후 1개월 내** (운영·SLA)

질문은 객관식 옵션 + `▶ 답:` 칸 형식. 결정 로그로 이관 가능한 구조.

권장 회수 일정:
- P0: D+10
- P1: D+31
- P2: feature freeze 직전

### Step 5 — 산출물 저장 (필수)

두 파일을 **동일 폴더**에 저장한다.

기본 경로 우선순위:
1. 사용자가 명시한 경로
2. 회의록/디자인의 상위 공통 폴더 (자동 추론)
3. `~/Desktop/{프로젝트명}/검수/{YYYY-MM-DD}/`

파일명 규약:
- `[{버전}]_{도메인}_비판노트.md`
- `[{버전}]_{도메인}_보강_질문지.md`

저장 후 채팅에 **두 파일의 절대 경로 모두** 보고.

### Step 6 — 실행 로그 (선택)

`~/proprium/agent-shared/skills/critique-reviewer/runs/log.md`에 한 줄 append:

```
YYYY-MM-DD | project={프로젝트} version={버전} | inputs={디자인N건, 회의록N건} | P0/P1/P2={n/n/n} | files={경로1, 경로2}
```

## 규약 (Conventions)

- **칭찬 금지**: 좋은 점은 한 줄도 쓰지 않는다. 사용자가 명시적으로 "균형 검토" 요청 시에만 예외.
- **구체 인용 필수**: 회의록 날짜·디자인 파일명 적시. *"5/15에는 'hourly mm 제거'"* 처럼.
- **단언형 문장**: "~할 수 있음", "~일 수도 있음" 같은 hedging 금지. 단언하고 근거를 댄다.
- **권고 디폴트는 디폴트일 뿐**: 동작 정책 표의 "권고"는 결정 가속용. 사용자 검토 필요함을 명시.
- **회의록 누락 없이 전수 Read**: 일부만 읽고 결론 내지 않는다. 폴더 깊이가 깊으면 재귀 탐색.
- **인증 실패 명시**: Figma 등 외부 리소스 접근 실패 시 분석에서 제외했음을 노트.
- **한국어 톤**: 엔터프라이즈/공공/금융 고객 맥락 가정. 캐주얼 톤은 비판 대상.

## 다른 스킬과의 연계 (예정)

- `doc-collect`: 회의록·기획서·디자인 자료를 먼저 수집 → 본 스킬의 입력으로 투입
- `jira-validation-triage`: 비판노트의 미해결 항목을 Jira 이슈로 변환 → 우선순위 매핑
- `figma-change-flow`: 보강 질문지의 P0 결정사항이 확정되면 Figma 노드 일괄 반영
- `kr-manual-change-tracker`: 동작 정책 변경 항목을 매뉴얼 변경 원장에 등록
- `planning-flow`: 보강 질문지 → 기획 작업 To-do 변환

연계 다이어그램은 `~/proprium/CRITIQUE_REVIEWER_MAP.md` 참조.
